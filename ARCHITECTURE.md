# Архитектура сервиса и MLOps-концепты

## Выбор архитектуры: Монолит

Для данного учебного проекта выбран монолитный подход.

| Критерий | Монолит | Микросервисы |

|----------|---------|--------------|

| Сложность кода | Низкая | Высокая |

| Деплой | Один контейнер | Оркестрация (K8s) |

| Масштабирование | Вертикальное/репликация | Горизонтальное |

| Подходит для | Прототип, MVP | Highload, много команд |

### Почему монолит
- Одна ML-модель и два эндпоинта
- Нет разделения на доменные зоны
- Проще тестировать и деплоить

### Когда переходить к микросервисам
- Несколько моделей с разными требованиями
- Разные команды разработки
- Независимое масштабирование predict/training
- Feature store, мониторинг дрифта

## Концепт: Брокер сообщений (RabbitMQ)

### Асинхронные batch-предсказания

[Клиент] -> [API] -> [RabbitMQ] -> [Worker] -> [Модель]
|
[Результат]

Клиент отправляет CSV, API возвращает `task_id`, Worker обрабатывает.

### Логирование через очередь

[API] -> [RabbitMQ] -> [Log Collector] -> [ELK Stack]


Не блокирует API, переживает пиковые нагрузки.

## MLOps-инструменты
### DVC (Data Version Control)
* Версионирование датасетов (как Git для данных)
* Переключение между версиями данных
* Хранение в S3/GCS, в Git - метаданные

### MLflow
* Tracking: параметры, метрики, артефакты экспериментов
* Registry: реестр версий моделей (staging/production)
* Serving: деплой модели как REST API

## Бизнес-метрики

### Expected Loss (ожидаемые потери)
EL = Σ [P(default) × LGD × EAD]
, где P(default) - из модели

LGD - доля потерь при дефолте (~60-80%)

EAD - кредитный лимит

Цель: минимизация EL при сохранении объёма кредитования

### Approval Rate с ограничением риска
Доля одобренных заявок при P(default) < порог.

Цель: максимизировать Approval Rate при том же уровне риска.

## ONNX-ML
Конвертация для ускорения инференса
```
python
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType

initial_type = [('float_input', FloatTensorType([None, 23]))]
onnx_model = convert_sklearn(model, initial_types=initial_type)
```
Преимущества: быстрее, кроссплатформенно, меньше образ.

### uWSGI + NGINX в production
Клиент -> NGINX (SSL, статика) ->  uWSGI (пул процессов) -> Flask

NGINX: прокси, SSL, буферизация

uWSGI: управление воркерами, перезапуск при падении
