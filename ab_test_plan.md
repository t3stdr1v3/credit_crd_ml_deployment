# План A/B-тестирования модели прогнозирования дефолта

## Цель

Сравнить две версии модели прогнозирования дефолта по кредитным картам:
- **Контрольная группа (A):** модель v1 - RandomForest
- **Тестовая группа (B):** модель v2 - GradientBoosting

## Дизайн эксперимента

### Разделение трафика
- Случайное распределение 50/50 на уровне запросов
- Параметр `?version=v1` или `?version=v2` в API-запросе
- В production-среде: балансировщик (nginx) направляет запросы к разным экземплярам сервиса

### Продолжительность теста
- **2 недели** для накопления статистически значимого количества наблюдений
- При потоке 1000 запросов/день: +-7 000 на группу

### Размер выборки
- Минимально: 5 000 наблюдений на группу
- Оптимально: 10 000+ наблюдений на группу


## Метрики оценки

### Основная метрика: F1-score для класса «дефолт» (1)

**Обоснование:** F1-score балансирует precision и recall, что критично для кредитного скоринга:
- Пропуск дефолта (низкий recall) -> прямые финансовые потери
- Ложное срабатывание (низкий precision) -> потеря потенциальных клиентов

### Дополнительная метрика: Recall

**Обоснование:** Для банка критичнее не пропустить дефолт, чем отклонить надёжного клиента. Recall показывает, какую долю реальных дефолтов модель обнаружила. Падение recall недопустимо даже при росте precision.

### Бизнес-метрики (расчётные)

1. **Expected Loss (ожидаемые потери):**
   - `EL = Σ(P(дефолт) × LGD × EAD)` для всех заявок
   - Снижение EL при том же уровне одобрения = успех

2. **Approval Rate (доля одобренных):**
   - Доля клиентов с P(default) ниже порога
   - Рост Approval Rate при том же уровне риска = успех

## Статистический анализ

### Тест: Z-тест для разницы пропорций

- Сравниваем F1-score двух моделей на тестовых данных
- **Нулевая гипотеза H0:** F1(v1) = F1(v2)
- **Альтернатива H1:** F1(v1) != F1(v2)
- **Уровень значимости:** alpha = 0.05

### Доверительные интервалы

- 95% ДИ для разницы метрик между группами
- Формула: `CI = (metric_v2 - metric_v1) ± 1.96 × SE_diff`
- Если ДИ не включает 0 → разница статистически значима

### Критерий успешности

| Результат | Условие |

|-----------|---------|

| **Успех v2** | F1-score v2 статистически значимо выше v1 **И** recall не упал более чем на 2 п.п. |

| **Ничья** | Разница в F1-score незначима → оставляем v1 (она проще и быстрее) |

| **Провал v2** | F1-score или recall значимо хуже → отклоняем v2, продолжаем с v1 |

## Техническая реализация

### Маршрутизация запросов через nginx

```
nginx
upstream model_v1 {
    server ml-api-v1:5000;
}

upstream model_v2 {
    server ml-api-v2:5000;
}

map $remote_addr $model_version {
    ~*[0-7]$ "v1";
    ~*[8-f]$ "v2";
}

server {
    listen 80;

    location /predict {
        if ($model_version = "v1") {
            proxy_pass http://model_v1;
        }
        if ($model_version = "v2") {
            proxy_pass http://model_v2;
        }
    }
}
```

## Логирование для анализа

model_version — версия модели

prediction, probability — результат

request — входные признаки

ground_truth — реальный исход (через месяц)
